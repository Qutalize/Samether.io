**バージョン**: v1.0
**作成日**: 2026-04-11
**前バージョン**: [v0.2](spec-v0.2-2026-04-10.md)
**ステータス**: ドラフト

## 1. 概要

Slither.io にインスパイアされた **位置情報連動型** リアルタイムマルチプレイヤーゲーム。
プレイヤーは現実世界を歩き回りながらサメを操作し、海中でエサ（小魚）を食べて成長・進化しながら他プレイヤーと競う。

---

## 2. 技術スタック

| レイヤー | 技術 |
|---|---|
| フロントエンド | Phaser + HTML5 Canvas + TypeScript（モバイルブラウザ対応） |
| バックエンド | Go + WebSocket |
| ビルドツール | Vite or Amplify |
| 位置情報 | Amazon Location Service（移動距離の算出・トラッキング） |
| インフラ | AWS (Fargate / ECS + ALB + ElastiCache) |
| DB | 最小構成（暫定1位のみ保持。Redis or インメモリ） |
| パッケージマネージャ | npm |

### インフラ方針

- リアルタイム通信が主要ボトルネックのため、WebSocket のスケーリングを最優先
- 負荷が問題になった場合、ALB で WebSocket サーバーを水平スケール
- Terraform は初期段階では不要。手動構築後、必要に応じて IaC 化

## 3. 操作システム

### 3.1 GPS モード（屋外）

- スマートフォンの GPS で現実世界の位置を取得
- 現実の移動がゲームマップ上のサメの移動に反映
- **移動距離がチャージポイント (CP) として蓄積される**（後述）
- 移動距離の算出には **Amazon Location Service** を使用
  - Tracker を用いてデバイスの位置を継続的にトラッキング
  - サーバー側で位置履歴を取得し、移動距離（メートル）を算出して CP に変換
  - 参考: https://aws.amazon.com/jp/location/

### 3.2 タッチ操作モード（室内）

- 画面上の指操作（スワイプ / バーチャルジョイスティック）でサメを動かせる

### 3.3 レーダー探索

- サメの前方に **扇形のレーダー** が点滅表示される
- レーダー範囲内のエサや他のサメを一時的に可視化する
- レーダーの点滅周期はサメの種族によって異なる

---

## 4. チャージポイント (CP) システム

現実世界での移動距離がゲーム内リソースに変換される仕組み。

| 項目 | 内容 |
|---|---|
| 蓄積条件 | GPS モードでの移動距離（メートル単位） |
| 蓄積レート | 1m 移動 = 1 CP |
| 用途 | ダッシュ、特殊能力の発動 |
| ダッシュ消費 | CP × サメ種別のスケール係数で移動距離が決まる |

- 攻撃系のサメ（ホオジロザメ等）は CP 消費効率が高い（少ない CP で長く走れる）が、**時間制限があるため大量の CP が必要** → 屋外で多く歩く必要がある
- 非攻撃系・深海魚系のサメは CP 効率は低いが、必要量も少ない
- ゲームプレイ中でも移動しながら CP を蓄積可能
- CP の算出方法: スタート地点とゴール地点を記録し、Amazon Location Service で移動距離を算出する

## 5. サメの進化システム

### 5.1 進化ツリー概要

ゲーム開始時に **3つのルート** から1つを選択し、各ルートの1段目のサメからスタートする。
全ルート共通で **5段階進化**。一定スコアに達すると次の段階に進化できる。

```
           攻撃系             非攻撃系            深海魚系
          ─────             ────────           ──────
  1段目  シュモクザメ         ドチザメ          ツラナガコビトザメ
            |                  |                    |
  2段目  イタチザメ          ネムリブカ           ノコギリザメ
            |                  |                    |
  3段目  アオザメ(最速)      シロワニ(燃費最高)     ラブカ
            |                  |                    |
  4段目  ホオジロザメ         ウバザメ          ゴブリンシャーク
            |                  |               (ミツクリザメ)
  5段目  メガロドン          ジンベエザメ        ニシオンデンザメ
        (攻撃系最上位)      (非攻撃系最上位)     (深海魚系最上位)
```

**スタート種の背景:**
- **シュモクザメ**: サメ界で最も個体数が多く、ヒエラルキーが低い。一度に約10頭生まれる
- **ドチザメ**: 小型肉食サメ。温和で非攻撃系の入口にふさわしい
- **ツラナガコビトザメ**: 小型の深海魚。弱いがロマン型ルートの出発点

### 5.2 攻撃系ルート

速度と攻撃力に特化。ただしエサの消費が激しく、時間制限がある。

| 種 | 段 | 特徴 |
|---|---|---|
| シュモクザメ | 1 | 攻撃系の入口。個体数が多くヒエラルキーが低い |
| イタチザメ | 2 | バランス型。速度がやや速い |
| アオザメ | 3 | **全サメ中最速**。燃費が悪い |
| ホオジロザメ | 4 | 大型の攻撃型サメ |
| メガロドン | 5 | 攻撃系最上位。巨大・高速・強力。**短命**（一定秒ごとにエサが必要、不足すると退化）。**血の知覚**: 他サメの方向を感知可能（レーダー不要） |

### 5.3 非攻撃系ルート

耐久力と効率に特化。安定したプレイスタイル。

| 種 | 段 | 特徴 |
|---|---|---|
| ドチザメ | 1 | 小型肉食。非攻撃系の入口 |
| ネムリブカ | 2 | やや遅いが燃費が良い |
| シロワニ | 3 | **超燃費型**（少量のエサで長時間維持可能。現実では1日サバ1匹で満足する） |
| ウバザメ | 4 | 大型。ジンベエザメ寄りの性能 |
| ジンベエザメ | 5 | 非攻撃系最上位。巨大でオーソドックス。安定した性能 |

### 5.4 深海魚系ルート（ロマン型）

進化途中は弱いが、最上位のニシオンデンザメが非常に強力。

| 種 | 段 | 特徴 |
|---|---|---|
| ツラナガコビトザメ | 1 | 小型で弱い。探索範囲がやや広い |
| ノコギリザメ | 2 | ノコギリで近接攻撃が可能 |
| ラブカ | 3 | 深海適応。探索範囲が広がる |
| ゴブリンシャーク (ミツクリザメ) | 4 | 射程の長い噛みつき攻撃 |
| ニシオンデンザメ | 5 | 深海魚系最上位。**探索範囲最大**、**エサをあまり必要としない**、**エサを食べなくても自動成長**。スピードは遅い。現実では約400歳の個体が確認されている最長寿のサメ |

**特殊アイテム**: マップ上に極稀に出現する特殊アイテムを取得すると、種族を問わず即座にニシオンデンザメに変身できる

### 5.5 特殊種

| 種 | 備考 |
|---|---|
| ネコザメ | 特殊なエサ「貝」のみ食べられる。ニッチなプレイスタイル |

※ コバンザメはサメではないため登場しない

### 5.6 サメの衝突ルール

- Slither.io 準拠 — **鼻（頭）が相手の体節に当たった方が死亡**
- 死亡時にはポイント放出やエフェクトなどのイベントが発生する

## 6. 領域システム（将来実装予定）

### 6.1 領域の形成

- サメが移動すると軌跡が残る
- 軌跡が **一次連結（閉じた形）** になると、囲まれた内部が **そのサメの領域** となる
- 領域はサメの移動に伴い動的に変化する

### 6.2 領域の強さ

- 領域の強さ = そのサメの種族ランク（進化段階）
- **弱いサメ** が **強いサメの領域** に入ると → **即死**
- **強いサメ** が **弱いサメの領域** に入ると → **無影響**
- 同格サメの領域 → 通常の衝突ルールが適用

### 6.3 バランス調整（強者抑制メカニズム）

強いサメほど領域が拡大しやすい収束問題への対策:

| 対策 | 内容 |
|---|---|
| 速度バランス | 弱いサメほど基本移動速度が速い（逃げやすい） |
| 燃費バランス | 強い攻撃系サメほどエサの消費が激しい（維持コストが高い） |
| 領域減衰 | 軌跡は時間経過で消える（古い部分から消失） |
| サメの生態準拠 | スピードが速いほど多くのエサが必要（現実のサメの生態と一致） |

## 7. 衝突・戦闘ルール

### 7.1 基本ルール（Slither.io 準拠）

- 自分の **鼻（頭の先端）** が他のサメの **体節** に衝突 → **自分が死亡**
- サメの鼻は弱点（現実のサメの生態と一致）
- 自分自身の体節には衝突しない

### 7.2 階級差がある場合

- 上位種であっても、鼻が相手の体節にぶつかれば死亡する（鼻は常に弱点）
  - 例: ホオジロザメの鼻がドチザメの体節に当たった場合、ホオジロザメが死亡する

### 7.3 死亡時の処理

- サメの全体節がエサに変換され、死亡地点周辺に散らばる
- プレイヤーはリスポーン画面に戻り、選択中のルートの1段目のサメから再スタート

## 8. エサシステム

### 8.1 エサの種類

初期実装では1種類（小魚）のみ。光っている静的オブジェクトとして表示する。
段階的に以下へ拡張する。

| エサ | 出現条件 | 効果 |
|---|---|---|
| 小魚 (S) | マップ上に自然発生 | +1 経験値 |
| 小魚群 (M) | マップ上にやや稀に自然発生 | +3 経験値 |
| 残骸 (L) | サメ死亡時に散らばる | +5 経験値 |
| 貝 | マップ上に稀に発生 | ネコザメ専用エサ |
| 特殊アイテム | マップ上に極稀に発生 | ニシオンデンザメ化 |

### 8.2 食事スタイル（将来実装予定）

初期実装では通常捕食のみ。将来的に以下を追加予定。

| スタイル | 対象種 | 挙動 |
|---|---|---|
| 通常捕食 | 全種共通 | 頭がエサに接触で取得 |
| 噛みちぎり | ホオジロザメ以上の攻撃系 | 他サメの体節を奪い取れる |
| 丸呑み | ジンベエザメ等の非攻撃系上位 | 前方広範囲のエサを一括吸収 |

## 12. ボットシステム（将来実装予定）

## 13. インフラ構成

Amazon GameLift の採用も検討対象。高負荷なマルチプレイヤー処理を委譲できる可能性がある。ブラウザベースの場合はサブミッション方式が適切。

```
[モバイルブラウザ]
       |
       ├── WebSocket (wss://)
       |         |
       |     [ALB]
       |         |
       |     [Fargate / ECS] × N台（Auto Scaling）
       |     ├── WebSocket サーバー (Go)
       |     ├── ゲームロジック
       |     └── ボット管理
       |         |
       |     [ElastiCache (Redis)] ← リーダーボード・セッション管理（最小構成）
       |
       └── GPS 位置情報
                 |
           [Amazon Location Service]
           ├── Tracker (デバイス位置のトラッキング)
           └── 移動距離算出 → CP 変換
```

### スケーリング方針

1. **初期から複数インスタンス + ALB 構成**で開始する。起動台数は CPU 使用率の実測値に基づいて決定する
2. **Auto Scaling の閾値を低めに設定**し、早めにスケールアウト（インスタンス台数を増加）する。スケールアウトには約20秒要するため、閾値を低く設定することで負荷急増時の遅延を吸収する
3. Redis はリーダーボード（暫定1位のみ）とセッション管理に使用
4. DB は基本的に不要（ランキングは暫定1位のみ保持）

### ボトルネック分析

本ゲームのサーバー処理は、衝突判定・ゲームループ・状態同期など **CPU バウンドな計算が中心** となる。そのため、スケーリング戦略は CPU の性能と台数を重視する。

| 負荷の性質 | 対策 |
|---|---|
| CPU バウンド（リアルタイム計算: 衝突判定、状態同期等） | インスタンスの台数・CPU 性能を増強 |
| I/O バウンド（データ読み書き、外部 API 呼び出し等） | DB チューニング、キャッシング（Redis）で高速化 |

### 15.3 負荷バランスに関する注意

- **対策**: Person 2 が Phase 1 の UI を早期に完了した場合、Person 5 の CP 関連フロント実装（GPS 座標取得 UI、スタート/ゴール UI）を引き取り、負荷を均等化する
